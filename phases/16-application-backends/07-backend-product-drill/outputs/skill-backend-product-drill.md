# Skill Drill — Backend Product Answer

## Use this structure
1. Clarify the product contract and one likely skew source.
2. Name 3 to 5 requirements, including one user-visible SLO.
3. Estimate traffic, fanout, and retention before drawing architecture.
4. Choose one deep dive based on the dominant risk.
5. Explain degraded behavior, not just the happy path.
6. Close with one trade-off and one redesign trigger.

## Self-check
- Did I make the read/write asymmetry explicit?
- Did I name what "success" means for the user?
- Did I tie the architecture to one or two concrete numbers?
- Did I say how the system fails and how we would notice?
- Did I avoid giving the same answer as the previous prompt?
